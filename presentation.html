🎯 Objectif d'AnyMesh

Prendre des sorties brutes de modèles génératifs et les transformer en assets 3D utilisables en production — sans intervention manuelle.

  • TRELLIS (Microsoft) – RunPod GPU
  • TripoSR – modèle open-source local
  • Stability AI SF3D – API cloud
  • Unique3D – worker Docker auto-hébergé

Interface unifiée — un seul point d'entrée pour les 4.

📦 Ce que le pipeline produit

  • Retopologie field-aligned (Instant Meshes)
  • Texture baking automatique (LSCM + KDTree)
  • Auto-LOD (4 niveaux, export ZIP)
  • GLB prêt production (Unity / Unreal)

Le problème — sorties IA non utilisables en l'état

Une sortie TRELLIS typique, avant tout traitement :

🧩 100 à 700 composantes disconnectées

Pas un mesh unique — une collection de patches géométriques indépendants. Chaque patch est non-fermé.

🕳️ Non-watertight

Des trous partout, par design de l'algorithme. La plupart des outils de traitement supposent un mesh fermé.

🔺 50 000 à 200 000 triangles chaotiques

Positionnés aléatoirement, sans edge flow. Impossible à rigger ou subdiviser proprement.

🗺️ UVs inutilisables

Absents ou ne correspondant plus à rien après traitement. Pas de texture applicable directement.

💡 Ces sorties ne passent pas dans Unity ou Unreal sans travail manuel. AnyMesh automatise ce travail.

4 providers intégrés dans une interface unifiée

4 providers — interface unifiée

🟠 TRELLIS (Microsoft)

  • Accès via RunPod Serverless GPU (A40/A100)
  • Meilleure qualité visuelle
  • Supporte plusieurs images en entrée (multi-view)
  • ⚠ Génère une scène très fragmentée (127 compos.) — problématique pour la retopo

🔵 TripoSR

  • Modèle open-source, GPU local
  • Rapide (~28s), watertight, mesh propre (1 compo.)
  • Qualité visuelle inférieure mais topologie plus utilisable
  • Meilleur aspect ratio (1.82 vs 2.31 TRELLIS)

🟢 Stability AI SF3D

  • API cloud — une seule image
  • Résultat watertight, 1 composante
  • Meilleur aspect ratio (1.43) — triangles proches de l'équilatéral
  • Bon compromis qualité / coût / simplicité d'intégration

🟣 Unique3D

  • Worker Docker auto-hébergé
  • Contrôle total, pas de coût variable
  • 23 composantes — intermédiaire
  • ⚠ Texture 512×512 seulement, ~17min de génération
💡 Mode multi-image TRELLIS : 3–4 angles (face + dos + profils) améliorent significativement la reconstruction des zones occultées sur une vue unique.

⚙️ Post-processing

Retopologie · Texture Baking · Auto-LOD · Analyse qualité

Retopologie — le défi sur les sorties IA

⚠ Problème 1 — Fragmentation TRELLIS

100 à 700 composantes séparées, toutes non-fermées. Instant Meshes produit des trous aux jonctions entre ces composantes (issue GitHub #78, non résolu).

⚠ Problème 2 — Sanitization impossible

pymeshfix ferme les trous mais reconstruit la géométrie autour des bords, ce qui déforme la forme originale.

Solution : si le mesh a >10 composantes → sanitization désactivée automatiquement.

✅ Ce qui fonctionne

Sur TripoSR et Stability (watertight, 1 composante) : triangles isotropes, edge flow propre, résultats excellents pour animation / subdivision.

🔧 Instant Meshes — choix technique

  • Disney Research / Wenzel Jakob et al., 2015
  • Remaillage field-aligned — qualité difficile à égaler en Python pur
  • Intégration : subprocess

🔄 Pipeline d'intégration

→ GLB entrant
→ Conversion en PLY temporaire
→ Instant Meshes CLI (C++)
→ Triangulation N-gons (PyMeshLab)
→ Export GLB propre
La fragilité (dépendance externe, parsing stdout) est le prix à payer pour cette qualité sans réécrire un algorithme de remaillage complet.

Texture Baking — high poly → low poly

⚠ Après retopologie : la géométrie est propre mais la texture est perdue. Les coordonnées UV originales ne correspondent plus à rien.

🎨 Pipeline en 5 étapes

1
LSCM unwrap — génère des UVs propres sur le low poly. Minimise la distorsion angulaire.
2
KDTree construit sur les vertices du high poly
3
Pour chaque vertex low poly → vertex high poly le plus proche → UV → couleur dans la texture originale
4
Rasterisation triangle par triangle, interpolation par coordonnées barycentriques
5
Export GLB low poly + UVs nouveaux + texture 1024×1024 embarquée

📊 Résultat concret

Un mesh low poly (3 000 faces) avec une texture 1024×1024 qui capture visuellement les détails du high poly original (50 000 faces).

Ratio qualité visuelle / coût GPU bien meilleur qu'un high poly direct.

💡 Honnêteté sur la précision

L'approche KDTree vertex-to-vertex est une approximation. Un raycasting perpendiculaire à la surface serait plus précis aux bords (Blender / xNormal font mieux). La valeur ici est l'automatisation en un clic.

🔬 Modes de diagnostic visuel

🌡️ Heatmap face quality

Aspect ratio par triangle — vert (équilatéral) → rouge (dégénéré). Diagnostic immédiat de la qualité de la retopo.

🔴 Boundary edges

Arêtes non-partagées en rouge — visualise les trous et bords ouverts du mesh.

🟡 Non-manifold edges

Arêtes partagées par 3+ faces en jaune — problème topologique qui casse beaucoup d'algorithmes.

📏 Hausdorff distance

Heatmap de l'écart géométrique entre deux meshes — valide qu'une retopo n'a pas trop déformé l'original.

Infrastructure, task system et choix techniques

Architecture technique

Navigateur
    │ REST API (axios)
    ▼
FastAPI  (VPS ~10€/mois, pas de GPU)
    ├── File de tâches async (thread workers)
    ├── Trimesh · PyMeshLab · SciPy · NumPy
    └── Instant Meshes (subprocess C++)
         │
         ├── RunPod Serverless
         │   └── GPU A40/A100 à la demande
         │       TRELLIS (~45s, ~0.05$/run)
         │
         └── Docker worker (VPS ou machine locale)
             └── Unique3D (dépendances CUDA)
Pourquoi cette archi : Le VPS ne paie que les opérations légères (retopo, baking). Les opérations GPU sont déléguées à RunPod — on paie ~45s de GPU par génération, pas un GPU en permanence.

⚡ Task System

  • Pool de workers threads (FastAPI)
  • Frontend poll /tasks/{id} jusqu'à completion
  • Timeout : 5 minutes
  • Retry réseau : ×3 avec backoff exponentiel

📦 GLB-First

  • Tous les fichiers stockés et servis en GLB
  • Imports OBJ, PLY, STL → convertis à l'upload
  • Un seul format côté frontend
  • Three.js charge les GLB natif avec textures embarquées

📊 Coût infrastructure

  • VPS : ~10€/mois (opérations CPU)
  • RunPod TRELLIS : ~0.05$/génération
  • Stability AI : facturation à l'usage
  • Unique3D : 0€ (auto-hébergé)

Ce que ce projet m'a appris

🤖 Sur les sorties IA

Les modèles sont très différents dans leurs sorties. TRELLIS = meilleure qualité visuelle mais pire topologie. TripoSR = meshes propres mais moins détaillés. Il n'y a pas de provider universellement meilleur — ça dépend de l'usage final.

🔄 Sur la retopologie automatique

Il n'existe pas de solution parfaite pour retopologiser des meshes non-watertight. Instant Meshes est excellent sur des meshes propres, inutilisable sur des scènes TRELLIS très fragmentées. Le vrai problème est en amont : des modèles génératifs qui produisent directement des meshes watertight.

🎨 Sur le texture baking

L'approximation KDTree est suffisante pour un usage web interactif. La précision d'un Blender est inutile quand l'objectif est une prévisualisation à 60fps dans un navigateur.

☁️ Sur l'infrastructure GPU

RunPod Serverless est une excellente solution pour accéder à des GPU haut de gamme sans infrastructure fixe. La latence de cold start (~10s) est le seul inconvénient notable.

Prochaines étapes identifiées

🔗 Re-baking post-simplification

La logique est implémentée pour la retopo, pas encore branchée sur la simplification. Le chaînon manquant évident du pipeline.

🎯 Raycasting pour le baking

Remplacer le KDTree vertex-to-vertex par un vrai raycasting perpendiculaire à la surface pour améliorer la précision aux bords.

🤖 Modèles génératifs watertight

Le vrai problème de la retopo sur TRELLIS se résoudrait en amont avec un modèle qui génère directement des meshes fermés.
